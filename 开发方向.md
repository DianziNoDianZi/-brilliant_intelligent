# Brilliant Intelligent — 开发方向文档

## 一、项目愿景

构建一个以**内在好奇心驱动**的自主智能体，使其能够在没有外部任务奖励的前提下，自发地探索环境、积累技能、并通过记忆回放与技能编译实现持续成长。最终目标是让智能体从抽象网格世界出发，逐步迁移到真实桌面应用、游戏场景，最终成为一个具备多模态感知、终生学习能力、并能与人类操作系统安全共处的通用操作代理。

核心理念可以概括为一条闭环：

> **预测误差 → 好奇心奖励 → 探索行为 → 情节记忆 → 后台回放 → 技能巩固 → 快回路编译**

这条闭环贯穿全部四个阶段，是整个系统的"生命线"。

---

## 二、技术路线总览

项目采用**四阶段渐进式演进**策略，每一阶段都建立在前一阶段已验证的闭环之上，复杂度梯度递增：

| 阶段 | 环境 | 核心能力 | 关键交付 |
|------|------|----------|----------|
| Phase 1 | 抽象网格世界 | 闭环自发运行 | 纯 Python 可视化脚本 |
| Phase 2 | 真实 UI 截图 | 视觉+语言理解 | 可听令操作简易应用 |
| Phase 3 | 实时游戏场景 | 双回路+技能生长 | 快慢回路并行系统 |
| Phase 4 | 多应用混合部署 | 多模态+终生学习 | 长期运行的自主体 |

每个阶段都必须通过明确的验证指标才能进入下一阶段，避免"在沙地上盖楼"。

---

## 三、分阶段开发方向

### Phase 1：最小闭环验证（网格世界，纯 Python）

**方向定位**：用最低成本验证核心算法闭环的可行性，不引入任何真实感知与执行复杂度。

**关键决策**：
- 环境用 `gymnasium` 或 `pygame` 实现，状态用向量表示，避免视觉处理负担。
- 世界模型采用轻量 LSTM（动作条件 RNN），输出下一状态分布，损失为 NLL 或 MSE。
- 内在奖励 `r_i = f(δ_t)`，使用滑动窗口归一化，保证奖励信号在不同阶段保持可比性。
- **区分两种不确定性（设计预留接口，Phase 1 后期或 Phase 2 实现）**：必须区分由环境固有随机性引起的预测误差与由智能体自身知识缺乏引起的预测误差，**只对后者给予奖励**。否则智能体会被环境随机噪声"卡住"，反复刷高误差却学不到东西（即"噪声电视"问题）。实现方式可选：
  - **系综分歧**：维护多个世界模型子网络，以它们对同一 (s, a) 预测的分歧作为好奇信号——分歧高说明模型尚未收敛（知识缺乏），分歧低但单模型误差高说明是环境固有随机性（不应奖励）。
  - **跨周期比较**：比较同一状态在不同学习阶段的预测改善量，改善量大者视为知识增长点。
  - **基于信息增益的探索奖励**：以模型参数后验更新量作为奖励，天然抑制纯随机噪声。
  - 网格世界阶段可不实现，但内在奖励模块必须预留 `uncertainty_type` 字段与可插拔的误差分解器接口，避免后期重构。
- 策略网络用 REINFORCE 或在线 Actor-Critic，奖励**完全来自内在好奇心**，不掺杂外部任务奖励。
- 情节记忆采用环形缓冲器，按 `δ + r` 优先级采样，在 episode 结束或探索步数达阈值时触发后台回放。

**验证重点**：
1. 早期行为是否为随机动作；
2. 好奇心曲线是否上升并聚焦到可交互元素（文件夹、按钮）；
3. 偶然成功（如双击打开文件夹）后，该行为频率是否快速增加；
4. 回放后技能是否趋于稳定。

**风险点**：纯随机探索可能长时间无法命中可交互元素，需要在动作空间设计上保证可交互元素密度足够，或在策略网络中引入轻微的探索偏置。

### Phase 2：视觉与语言界面理解（扩展到真实 UI 截图）

**方向定位**：将闭环从抽象向量迁移到真实视觉感知，并引入语言规划作为"慢回路 V0"。

**关键决策**：
- 视觉前端使用预训练轻量模型（MobileViT），顶部加可在线适应的检测头，输出 UI 元素边界框与类别。
- OCR 优先选用 PaddleOCR（成熟、中文友好），输出 `{文本内容, 位置, 所属实体}` 结构。
- 状态表示从原始像素升级为 **世界状态图谱（WSG）**——融合视觉检测与 OCR 的 JSON 结构，包含 `entities` 与 `relations`。世界模型的预测对象从像素转向 WSG 的变化，大幅降低预测维度。
- 语言规划核心使用开源 LLM（Qwen-2.5），通过 vLLM 或 API 部署，提示词模板将 WSG + 任务指令转换为结构化子目标计划，每个子目标映射到 WSG 实体 ID。
- 引入**内模拟**：世界模型接收动作序列后预测 WSG 变化并给出置信度，低置信度步骤触发真实单步执行验证。

**验证重点**：
1. WSG 生成质量（实体召回率、关系准确性）；
2. LLM 子目标计划与 WSG 实体的对齐率；
3. 沙盒环境（Windows VM/容器）中计算器、记事本的自然语言指令执行成功率。

**风险点**：OCR 在小字号、低对比度场景下不稳定；LLM 输出的子目标可能引用不存在的实体 ID，需要在提示词中加入实体枚举约束与校验层。

### Phase 3：双回路与实时技能生长

**方向定位**：从慢思考中蒸馏出快速反应通路，实现"操作与思考并行"，进入实时交互领域。

**关键决策**：
- **快回路**：小型前馈/浅层循环网络（1D CNN/MLP），输入感知特征压缩投影，直接输出连续动作参数。训练侧 Python，推理侧导出 ONNX/TorchScript，由 Rust（tract/ort）或 C++（TensorRT/ONNX Runtime）加载，通过管道或共享内存与 Python 通信。
- **意图向量接口**：慢回路输出包含主目标、操作风格参数、警戒触发规则的意图向量，通过 ZeroMQ 或共享内存传递给快回路，不打断其循环。
- **技能自动编译**：当某子目标序列成功率超过阈值（如 95%），收集多条成功轨迹，用行为克隆训练快回路网络，并将序列抽象为宏动作存入技能库。编译后慢回路只下发技能名。
- **慢回路异步化**：作为独立进程进行长时间深度推理，异步更新意图向量，快回路轮询最新意图。

**验证重点**：
1. 快回路稳定操作频率 > 30 FPS；
2. 慢回路战术决策日志的合理性；
3. 多局后胜率/任务完成时间的改善趋势；
4. 技能编译触发后，慢回路负载是否显著下降。

**风险点**：快慢回路状态同步可能出现竞态；技能编译过早会导致快回路过拟合到特定场景。需要明确编译阈值与回滚机制。

### Phase 4：多模态、终生学习与系统整合

**方向定位**：赋予完整感官，实现跨应用迁移和长期自主成长，并建立不可绕过的安全边界。

**关键决策**：
- **流式音频前端**：Rust（cpal + hound）或 C++（PortAudio）实现最小延迟采集，STFT 在采集端完成，特征向量经共享内存送入 Python 世界模型。世界模型增加音频预测头，利用视听同步作为自监督信号。
- **系统事件感知**：Rust（windows-rs）或 Python（psutil, win32api）采集 CPU/内存/网络/通知，作为特殊节点注入 WSG。
- **层级记忆**：三层结构——原始体验 / 技能块（应用特定）/ 抽象操作模板（设备无关，如"保存文档"）。新应用场景下，慢回路基于抽象模板搜索记忆，匹配后仅微调定位参数。
- **终生学习**：弹性权重巩固（EWC）在线版本防遗忘；或动态稀疏训练（rigl/SET）允许网络成长。
- **睡眠式巩固循环（终生学习的关键机制）**：在系统空闲时段自动触发，**不仅是回放高价值记忆**，而是让系统在离线状态下"做梦"，包含三种具体形式：
  - **生成式回放**：用世界模型基于记忆中的状态，生成新的"想象轨迹"（imagination rollouts）来训练快回路。这使快回路能在不占用真实环境时间的情况下，从世界模型的预测中获得大量训练样本，加速技能编译。
  - **技能重组**：将不同上下文中学到的片段重新组合成新的技能序列。例如把"打开文件"和"查找替换"两个独立技能组合成"批量处理"新技能。重组后的序列若在内模拟中验证通过，即作为候选技能入库。
  - **跨应用抽象**：将"文件→保存"这类操作从具体应用（记事本、Word）中剥离为设备/应用无关的抽象操作模板，存入层级记忆的最顶层。下次面对新应用时，慢回路只需匹配抽象模板并微调具体定位参数。
  - 这三种形式共同构成系统的"离线梦境"，是能力能否持续增长而非饱和的关键。睡眠式巩固的触发频率、生成轨迹质量、重组成功率必须纳入可观测仪表盘。
- **安全沙盒**：三层强制隔离——AI 决策层 / 权限防火墙（Rust 实现，硬编码禁止规则）/ 真实操作系统。防火墙作为快回路指令的最后关卡，保证不可绕过性。

**验证重点**：
1. 灾难性遗忘指标；
2. 新任务上手速度；
3. 技能编译率；
4. 跨应用迁移步数；
5. 长期运行稳定性。

**风险点**：EWC 与新任务学习的权衡；安全防火墙规则需要随能力扩展持续更新，否则会成为能力天花板或被绕过的薄弱点。

## 四、核心技术决策与架构原则

### 4.1 接口先行

模块间使用结构化数据格式（JSON/Protobuf）和通信中间件（ZeroMQ/gRPC/共享内存）。任何模块的语言替换（Python → Rust/C++）不得影响整体接口契约。这是后续"渐进式替换"能落地的前提。

### 4.2 可观测性优先

为以下"生命体征"建立仪表盘，开发阶段持续监控：
- 世界模型预测误差曲线
- 内在奖励分布与滑动统计
- **不确定性类型分布**（环境随机性 vs 知识缺乏的占比，用于诊断是否陷入"噪声电视"）
- 记忆回放频率与优先级分布
- 技能编译次数与成功率
- **睡眠式巩固触发频率、生成轨迹质量、技能重组成功率**
- 快慢回路延迟与吞吐

没有可观测性，闭环是否真正"活着"将无法判断。

### 4.3 渐进式替换

性能敏感模块（快回路、音频前端、防火墙）先在 Python 中完成功能原型，验证算法正确性后再按需迁移至 Rust/C++，**接口保持不变**。禁止直接用底层语言做原型，避免调试成本与算法迭代速度下降。

### 4.4 测试环境复杂度梯度

严格遵循梯度递增，每一步通过验证后才进入下一级：

```
网格世界 → 简单桌面应用 → 单个游戏 → 多应用混合操作
```

跨级跳跃是项目失败的主要风险来源。

## 六、里程碑与交付物

- **M1（Phase 1 完成）**：纯 Python 脚本，可视化展示智能体在网格世界中从随机动作到聚焦可交互元素、偶然成功后行为频率上升、回放后技能稳定的全过程。
- **M2（Phase 2 完成）**：可听令操作计算器/记事本的智能体，纯 Python 实现，自然语言指令可执行。
- **M3（Phase 3 完成）**：双回路系统在简化 RTS/俯视角动作游戏中稳定运行，快回路 > 30 FPS，技能编译可观测。
- **M4（Phase 4 完成）**：受控 VM 中长期运行的自主智能体，具备多模态感知、跨应用迁移、终生学习能力与不可绕过的安全边界。

---

## 七、贯穿始终的开发准则

1. **接口先行**——任何模块替换不影响整体。
2. **可观测性**——生命体征仪表盘贯穿全周期。
3. **渐进式替换**——Python 原型验证算法，Rust/C++ 优化性能。
4. **复杂度梯度**——网格 → 桌面应用 → 单游戏 → 多应用混合，逐级通过。
5. **闭环优先**——任何阶段，"预测误差→好奇→记忆→技能"闭环必须可运行、可观测、可验证，再谈扩展。
